GRU-based Lag-Informed Decision Engine for GPU Job Dispatching

Abstract

Service-Lag-Aware Dispatching Under Fixed Placement

Modern GPU clusters must balance latency, slowdown, fairness, and resource fragmentation. Existing GPU cluster schedulers often rely on heuristic dispatching rules such as FIFO, shortest-job-first, slowdown priority, and resource-aware scheduling. These rules are strong and practical, but they cannot explicitly adapt their decisions to queue pressure, resource availability, and accumulated service imbalance.

GLIDE studies whether reinforcement learning can improve GPU job dispatching under a fixed placement backend. It uses a service-lag-aware GRU actor-critic scheduler trained with PPO in a trace-driven GPU cluster simulator. The policy observes a bounded visible queue window, filters infeasible jobs, and selects one feasible job to dispatch, while First-Fit placement remains fixed.

This project does not claim state-of-the-art GPU scheduling performance. Instead, it focuses on a controlled dispatching comparison: all evaluated methods share the same trace split, node subset, placement backend, and evaluation metrics.

Key Idea

Learn Dispatching, Keep Placement Fixed

GLIDE separates dispatching from placement. The reinforcement learning policy only decides which feasible job should be launched from a bounded visible queue window. The low-level node allocation is handled by the same First-Fit placement backend for all methods.

This design makes the comparison clean: performance differences should come from dispatching order, not from different placement implementations.

Motivation

Why GPU Job Dispatching Is Hard

Strong Heuristic Baselines

Simple dispatching rules such as SJF and slowdown priority already perform well on conventional latency and slowdown metrics. A learned policy must improve meaningful fairness-oriented behavior without degrading scheduling efficiency.

Fairness-Efficiency Trade-off

Reducing service imbalance is not useful if the scheduler achieves it by delaying jobs, increasing tail latency, or leaving jobs unfinished. GLIDE therefore evaluates fairness together with latency, slowdown, and completion reliability.

Event-Driven Partial Observation

Scheduling decisions occur only when jobs arrive, complete, or resources become available. The scheduler observes current jobs and resources, but not future arrivals or completions. This motivates recurrent state modeling.

Method

Method Overview

01

State: Queue Pressure + Lag Context

GLIDE observes per-job features for the visible queue plus global summaries: queue length, wait and slowdown statistics, GPU utilization, fragmentation, feasible-job ratio, recent debt gaps, and CFS normalized service-lag summaries by GPU demand group.

02

Action: Feasible Dispatch Only

At each event, the simulator exposes a bounded visible pool, filters jobs that First-Fit can actually place, samples up to 32 feasible candidates, and adds an explicit pass action. The policy never chooses an impossible placement.

03

Policy: Set Encoder + Event Memory

A multi-head queue-set encoder summarizes the visible waiting jobs. A GRU carries memory across scheduling events. The actor scores each candidate using job features, recurrent context, candidate-set mean and max summaries, and job-context interactions.

04

Reward: Move Lag Down

The reward tracks the maximum positive CFS normalized lag among active GPU groups. It penalizes log lag level and log lag growth, rewards log lag reduction, adds a pass penalty, and attaches terminal gate costs for efficiency and unfinished jobs.

The novelty is not a new PPO variant. The contribution is a service-lag-aware RL dispatching framework that combines fixed-placement evaluation, feasible-action filtering, queue-aware recurrent modeling, efficiency-gated checkpoint selection, and behavior diagnostics.

Implementation Highlights

What Is Actually Inside GLIDE?

Reward is based on lag motion, not a final score only

During each scheduling macro-step, GLIDE computes a CFS-style lag potential before and after the interval. The potential is the largest normalized positive service lag among GPU demand groups currently waiting or running.

r = -0.05 log(1 + lag_after) - growth + reduction - pass_cost - gates

The log transform keeps extreme lag values from dominating PPO updates. Growth and reduction are separated so the policy gets directional feedback: making lag worse is not the same signal as failing to improve it.

The action set is deliberately constrained

GLIDE first looks at the Top-K visible queue, then checks feasibility using the same First-Fit placement backend used by all baselines. The policy scores only feasible candidates plus pass. This keeps the RL problem focused on dispatch order, not learning a hidden placement solver.

  • Visible pool: 32 jobs
  • Candidate sampling: uniform_pool
  • Mean feasible jobs: 3.09
  • Mean action-set size with pass: 4.09

Selection is guarded by efficiency gates

Checkpoints are not selected by fairness alone. The validation score is CFS lag AUC plus penalties if slowdown, average wait, P95 wait, fairness floor, or unfinished jobs violate reference gates. This prevents a seemingly fair policy from winning by simply slowing the cluster down.

  • Best update: 33
  • Gate penalty at selected checkpoint: 0.0
  • Unfinished jobs: 0
  • Pass ratio: 0.0%

Evaluation

Controlled Trace-Driven Evaluation

  • Dataset: Alibaba Cluster Trace GPU v2023
  • Training split: 1000 jobs from offset 0
  • Evaluation split: 1000 jobs from offset 2000
  • Node subset: 64 nodes
  • Placement backend: First-Fit for all methods
  • Visible queue window: Top-K = 32
  • Candidate sampling: uniform_pool
  • Candidate count: 32
  • Policy model: GRU actor-critic
  • Training algorithm: PPO with GAE
  • Primary selection metric: CFS Normalized Service-Lag AUC
All methods are evaluated under the same trace window, node subset, placement backend, and metrics. This isolates whether reinforcement learning improves dispatching order itself.

The improvements are moderate, but they are consistent across service-lag, average latency, slowdown, and tail-related metrics. This is meaningful because SJF and slowdown-priority are already strong heuristic baselines.

GLIDE frequently selects the highest-risk feasible job within the visible action set. The pass action is available, but the selected checkpoint never uses it. This indicates that the improvement is not caused by artificial idling. The learned behavior is closer to a lag-aware risk prioritizer than to a scheduler that waits for a better future placement.

Training

Training Dynamics

The selected checkpoint occurs at update 33. Validation CFS lag AUC drops from 12311.41 at update 0 to 13.74 at update 33. Later checkpoints drift upward, and the final checkpoint performs substantially worse.

This shows that GLIDE can learn an effective dispatching policy, but PPO training is unstable in this event-driven trace-based scheduling environment. Because the reward is shaped from lag movement while checkpoint selection uses gated validation metrics, we report the validation-selected checkpoint rather than the final checkpoint.

Limitation: PPO checkpoint drift remains an important issue. Future work may explore more stable constrained RL, offline-to-online training, or improved early stopping.

Contributions

Research Contributions

01

Formulate service-lag-aware GPU job dispatching as an event-driven RL problem under a fixed placement backend.

02

Propose GLIDE, a GRU-based PPO dispatching policy with feasible-action filtering and service-lag-aware reward shaping.

03

Design a controlled trace-driven evaluation protocol that compares learned dispatching against strong heuristic baselines under the same First-Fit placement backend.

04

Use CFS normalized service-lag AUC together with efficiency-gated checkpoint selection to evaluate fairness-oriented improvements without severe latency or slowdown degradation.
